Front-End Is Way Harder Than Back-End

Developing web sites with HTML, CSS and templating languages like Liquid and writing programs — in Ruby, Python, PHP etc. — is not the same thing, at all.

Programming languages are older and more advanced than markup and styling languages. They focus on data, not humans as in case of web design and development.

Manipulating code bits living in an abstract but uniform environment — like the computer — is easier than manipulating pixels living in the perception of humans, millions of them and all different.

Code and data runs in standard virtual machines when front-end results run on displays which vary from mobile handsets to large projector screens, and, are rendered by many different browsers.

Coding web sites is like shooting on a moving target; back-end programming, well, on the other hand is stable, testable, more straightforward.

Tools And Best Practices For Front-End Developers

Yes, you can have them too.

You'll never have that advanced ecosystem a full-featured programming language is equipped with: automated tests, intelligent debugger & performance analyser, code coverage tool, documentation generator, frameworks built on frameworks, and all.

But you can re-use the back-end principles of creating good code in front-end development too, and, you can adapt as well some of the tools back-end developers use.

A good starting point for analysis could be the Ruby programming language and the Ruby on Rails web development platform, probably the best tools of the moment for creating web applications:

1. Human Friendly Code

Ruby was created / optimised for developers, not the machine.

Programming with Ruby is very straightforward today, the language constructs are so close to plain English even a child can understood and use them.

Ruby code needs no documentation — it is so elegant it documents itself.

—

As a first lesson try to create human friendly, readable, understandable design code which documents itself.

Make your design your friend. Changing, editing, reviewing, enhancing, extending a Ruby code is always a joy. In contrast, your HTML or CSS code will suck after a while even for you, the creator.

The biggest challenge in web front-end development is to integrate, by innovative and meticulous naming techniques the HTML, CSS and Javascript elements into a seamlessly co-working entity.

Mark your code in a straightforward, easily rememberable way which works in any context assuring the flow. So you don't have to look up different source files to find out how to style, or add interaction to an element.

2. Don't Repeat Yourself

Scalability is everything, you know.

This principle turned out to be a very strong building block of the success of the Rails framework.

Rails was created by substracting the essence from a well known, top notch web application (Basecamp) created with Ruby. All best practices were turned into reusable modules forming the framework itself.

Basecamp, thus Rails was continuously refactored and optimised until all fat was trimmed. The outcome was the simplest and most robust tool which can be reused, again, to create more successful applications in a faster and easier way.

Smart people are maximising the outcome of every work they done. Beside the final product there is always something more: an enhanced tool— and mindset helping them to do better on the next project.

They don't repeat themselves. Instead the tool, the framework, the ecosystem they have extracted repeats and embeds itself automagicaly in every following project they create.

Web front-end development is lacking tools and frameworks back-end developers are equipped with. Industry best practices are not extracted into a common knowledge base and then enhanced by a community. Designing for web is not scalable.

3. Test Driven Front-End Development

I know it sounds impossible. No test tool can see as human eyes do.

But if back-end developers can do it maybe the front-enders will be able to do so.

In an ideal scenario — you have human friendly design code and a powerful design framework built on your best knowledge — a strong HTML/CSS/Javascript base is at your disposal whenever you start a new project.

This framework in order to be perfect must be tested on various browsers, operating systems, display sizes, color depths and so on. No testing tool will warn you on bugs like this.

And the tl;dr

Front end developers (HTML/CSS programmers) should learn from their back end developer brothers (coding in Ruby, Python etc.) to make their everyday work more enjoyable, more bullet-proof and widely sustained.

The current front-end developer elite shares some thoughts on best practices, maintains a few boilerplate templates but there are no frameworks and ecosystems as in back-end development empowering both its creators and users.

Both principles and tools can be borrowed / adapted from back-end developers — what we only need is a charismatic leader to lead the process and become a hero.
